# Static Catalog Example

This example shows how to use `wv catalog import` for offline query development and CI
validation. The import saves your database's table schemas as Wvlet `type` definitions in the
`catalog/` folder, so queries compile without a database connection.

See the full documentation: [Importing Database Catalogs](https://wvlet.org/wvlet/docs/usage/catalog-import).

## Setup

Import the table schemas from your database once:

```bash
# Default profile (DuckDB)
wv catalog import

# A profile from ~/.wvlet/profiles.json, limited to one schema
wv catalog import --profile production --schema sales
```

This writes one file per schema:

```
catalog/
  mydb/
    sales.wv        # type defs for tables in mydb.sales
```

```wvlet
-- catalog/mydb/sales.wv
-- Generated by `wv catalog import`. DO NOT EDIT.
type orders in mydb.sales = {
  order_id: long
  status: string
  total_amount: decimal[10,2]
}
```

## Write queries against the imported schemas

```wvlet
-- sales_summary.wv
from sales.orders
where status = 'completed'
group by status
select status, count(*) as order_count
```

Compile offline — the reference resolves through the imported type and produces
schema-qualified SQL:

```bash
wv compile -f sales_summary.wv
```

## Keeping the catalog updated

Re-running the import overwrites the generated files deterministically, so `git diff` shows
exactly what changed in the database:

```bash
#!/bin/bash
# update-catalog.sh
wv catalog import --profile production
if git diff --quiet catalog/; then
  echo "No catalog changes"
else
  git add catalog/
  git commit -m "Update catalog snapshot $(date +%Y-%m-%d)"
  git push
fi
```

## CI validation

```yaml
# GitHub Actions example
- name: Validate queries
  run: |
    for query in *.wv; do
      wv compile --strict -f "$query"
    done
```

## Best practices

1. **Commit the catalog folder** together with your queries, like a lockfile
2. **Do not edit generated files** — add hand-written `type`/`model` definitions in your own
   `.wv` files, which a re-import never touches
3. **Schedule regular imports** (cron or CI) to catch schema drift early
